## Context

Auth currently validates against `Auth:Users` in `appsettings.shared.json` — a static list with plaintext passwords. The login endpoint (FastEndpoints `LoginEndpoint`) reads this list directly. The Blazor login page (from the blazor-ui change) will do the same. Moving to a DB-backed user store requires changing the validation path in both, adding a `users` table, and adding admin endpoints + Blazor pages to manage accounts.

The system is closed — no self-registration. Only an Admin can create accounts. This means we need at least one Admin account to bootstrap. The seed strategy handles this without requiring a separate migration or admin CLI.

## Goals / Non-Goals

**Goals:**
- Persist users in PostgreSQL with PBKDF2-hashed passwords
- Admin-only REST endpoints for user CRUD
- Admin-only Blazor pages for user list and create
- First-run seed of an Admin account from config
- Login (both JWT and cookie paths) validates against DB

**Non-Goals:**
- No self-registration
- No password reset flow or email verification
- No edit-user or change-password page (delete + recreate)
- No OAuth / external identity providers
- No user profile data beyond username and role

## Decisions

### D1: PBKDF2 via `PasswordHasher<UserEntity>`

ASP.NET Core ships `Microsoft.AspNetCore.Identity.PasswordHasher<T>` in the base framework (no Identity package needed). It uses PBKDF2-SHA512, 100k iterations. No extra NuGet dependencies — just `services.AddSingleton<IPasswordHasher<UserEntity>, PasswordHasher<UserEntity>>()`.

**Alternative considered**: BCrypt.Net. Requires a third-party package; PBKDF2 via the built-in hasher is already hardened and avoids the extra dependency.

### D2: UUID primary key for users

User IDs are UUIDs (`Guid`) generated by the application, not auto-increment integers. This avoids leaking sequential IDs in API responses and delete endpoints.

### D3: First-run seed from config, DB is then authoritative

On startup, `UserSeeder.SeedAsync()` checks if any user with Admin role exists in the DB. If none, it reads `Auth:Users` from config, finds the first Admin entry, and inserts it with a hashed password. Subsequent startups skip seeding. Config `Auth:Users` is ignored at runtime after the first run.

**Why not always seed from config?**: If an admin changes their password via future tooling, a config-driven seed would overwrite it on restart. DB-first after initial seed avoids this.

### D4: Three admin REST endpoints, JWT-protected

- `GET /api/admin/users` — list all users (id, username, role, createdAt; no password hash)
- `POST /api/admin/users` — create user (username, role, password); returns 201 with user id
- `DELETE /api/admin/users/{id}` — delete user by UUID; returns 204

These follow the same pattern as the existing book admin endpoints. They require `[Authorize(Roles="Admin")]`.

### D5: `AdminUserService` in Blazor mirrors `AdminBookService`

The Blazor admin pages call the REST endpoints via `AdminUserService` (scoped), which reuses the JWT obtained by `AdminBookService`'s self-auth pattern. To avoid duplicating the JWT self-auth logic, extract it into a shared `AdminApiClient` that both services use.

**Alternative considered**: Direct DB access from Blazor pages. Works since Blazor Server is in-process, but bypasses the REST layer and duplicates auth checks. Keeping the REST boundary makes the admin actions auditable through the same endpoint logging.

### D6: No edit-user page

Changing a password requires delete + create. This keeps the UI simple and avoids a separate `PATCH /api/admin/users/{id}` endpoint. The admin can see the create date but not the old password, so this is the correct flow anyway.

## Risks / Trade-offs

- **Bootstrap lockout**: If the config seed entry has a wrong password or is missing, the first-run admin account won't be created and no one can log in. → Mitigation: `UserSeeder` logs a warning if seeding is skipped due to missing config, and the app throws on startup if no Admin exists in DB and no seed entry is found.

- **Deleting the last Admin**: An admin could delete their own account, locking everyone out. → Mitigation: The delete endpoint checks that at least one other Admin exists before proceeding; returns 409 if the deletion would leave zero admins.

- **`AdminApiClient` JWT cache**: The JWT obtained for admin API calls is cached in a scoped service (per Blazor circuit). If the token expires mid-session (default 8h), calls will fail silently. → Acceptable for now — session lifetime is short relative to expiry.

## Migration Plan

1. Add EF Core migration to create `users` table (UUID pk, username unique, password_hash, role, created_at)
2. Add `UserSeeder` to app startup (runs after `MigrateDbAsync`)
3. Update `LoginEndpoint` to validate against DB
4. Add three admin user endpoints
5. Add `AdminUserService` + shared `AdminApiClient` (or update `AdminBookService` to share the token)
6. Add Blazor pages (`Users.razor`, `CreateUser.razor`)
7. Update admin nav to include Users link

**Rollback**: Drop the `users` table and revert `LoginEndpoint` to config-based validation. Config `Auth:Users` still present so rollback restores full auth.
