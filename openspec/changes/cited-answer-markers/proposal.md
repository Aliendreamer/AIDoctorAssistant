## Why

The `ui-visual-redesign` change promoted citations into a visible reference apparatus — a numbered
source list beneath each answer — but deliberately deferred **inline `[n]` markers in the answer
prose**, because those require a change to how the RAG pipeline generates answers, not just CSS.
This change adds them: a physician reading a claim can see exactly which indexed source supports
it, tying each sentence to the numbered source list.

## What Changes

Inline numbered citation markers, generated by the model and rendered as a superscript that maps to
the existing source list.

**Answer generation (RAG)**
- Number the retrieved excerpts fed to the model — `[1]`, `[2]`, … — in the same order the
  `sources` list is built, so marker `n` always maps to `sources[n-1]`.
- Add a citation instruction to the three book-RAG system prompts (the shared base prompt used by
  Disease / Symptoms / Treatment, plus the GlobalSearch and DifferentialDiagnosis overrides): keep
  the existing "explain like a colleague" prose style, but **append** `[n]` after a claim the cited
  excerpt supports. Markers augment the prose; they don't replace the readable style.
- The web paths keep working: `EnrichWithWebAsync` is told to preserve any existing `[n]` markers
  unchanged (book sources keep indices `1..K` in the merged `[..book, ..web]` list, so preserved
  markers stay correct); web sources continue to be cited by article title. Pure-web answers
  (`AnswerFromWebAsync`) carry no markers.

**Rendering (Query UI)**
- Parse the answer text and render `[n]` / `[1, 2]` as styled superscript `.cite` markers with a
  tooltip naming the source.
- **Range-guarded**: a marker whose number is outside `1..sources.Count` renders as plain text —
  never a broken or mis-pointing link. Guards against model over-citation / hallucinated numbers.
- **Graceful degradation**: an answer with no markers renders exactly as today; a history-loaded
  answer (whose per-message sources aren't held in session) shows markers as plain text.

## Capabilities

### New Capabilities

- `answer-citation-markers`: model-emitted inline `[n]` citation markers, their guaranteed mapping
  to the source list, and their range-guarded rendering.

### Modified Capabilities

- None behaviourally re-specified here. This builds on `blazor-query-ui` (answer rendering) and the
  RAG answer-generation behaviour; existing requirements there are unchanged, only extended.

## Impact

- `MedAssist.AI/Plugins/RagPluginBase.cs` — number the context excerpts in `BuildResultAsync`; add
  the citation rule + updated example to the base `GetSystemPrompt`.
- `MedAssist.AI/Plugins/GlobalSearchPlugin.cs` — add the citation rule to its prompt.
- `MedAssist.AI/Plugins/DifferentialDiagnosisPlugin.cs` — add the citation rule + update the
  few-shot example to show markers.
- `MedAssist.Web/Services/QueryService.cs` — instruct `EnrichWithWebAsync` to preserve `[n]`
  markers.
- `MedAssist.Web/Components/Pages/Query.razor` — parse answer text; render `.cite` markers
  (range-guarded, degrades gracefully).
- `MedAssist.Web/Components/Pages/Query.razor.css` — `.cite` marker styling.
- `MedAssist.Tests` — a `MarkdownStripper` test asserting `[n]` survives stripping; a parser test
  for the marker segmentation + range guard.

No new dependencies. No schema or endpoint changes. No change to which sources are retrieved.

## Out of scope

- Persisting per-message sources so history-loaded answers keep clickable markers (a separate data
  change). Here, reloaded answers show markers as plain text.
- Numbered markers for pure-web answers.
- Clicking a marker to scroll/highlight the source row — v1 markers are visual + tooltip only.
